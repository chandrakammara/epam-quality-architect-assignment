# Automation Coverage Summary — Traceability Matrix

**Scope:** Part, Part Category, Part Parameters/Templates, Part Templates/Variants, Part Revisions, Part Test Templates (same scope as `docs/Requirements_And_Coverage_Summary.md`).
**Sources:** `docs/Requirements_And_Coverage_Summary.md` (business-area → manual-case mapping, §3–5), `docs/API_Schema_Analysis.md` (endpoint inventory and recommended automation tiers, §10), and a direct read of every file under `automation/ui` and `automation/api` as of this document's creation.
**Principle:** this matrix states only coverage that could be confirmed by reading the automation source directly. Nothing is inferred from intent, naming, or partial similarity — a gap is marked **Gap** rather than guessed at.

## 1. How to read this matrix

| Coverage Status | Meaning |
|---|---|
| ✅ **Automated** | The requirement's core, documented behaviour has an executing automated test (UI and/or API) that exercises it end-to-end. |
| 🟡 **Partial** | Some slice of the requirement is automated (named explicitly in Notes); the remainder is manual-only today. |
| 🔴 **Gap** | No automated test executes any part of this requirement; manual coverage may still exist per the RTM. |
| ⚪ **Scaffolded, not executed** | Supporting automation code exists (e.g. a service class) but no test file currently invokes it — deliberately distinguished from both "Automated" and "Gap" since the code is present but unproven. |

Manual test IDs are quoted directly from `docs/Requirements_And_Coverage_Summary.md` §3 (itself sourced from `test-cases/InvenTree_Parts_QA_Test_Cases.xlsx`, marked `Status = Draft` / not yet execution-verified in that workbook). UI automation is drawn from the 9 spec files under `automation/ui/tests/`; API automation is drawn from the 4 spec files under `automation/api/tests/` (`auth.spec.js`, `part-crud.spec.js`, `part-validation.spec.js`, `part-search-pagination.spec.js`).

## 3. Summary

- **Fully automated end-to-end today:** Part core CRUD (REQ-007, both layers) and the REQ-010 cross-functional UI journey.
- **Partially automated:** REQ-001, REQ-003 (UI only), REQ-004, REQ-005 (UI only), REQ-008 (API only), REQ-009.
- **Complete gaps (no automation on either layer):** REQ-002 (detail tabs / special endpoints), REQ-006 (templates, variants, revisions).
- **Scaffolded but unproven:** Category API automation (`CategoryService.js` + fixture wiring exist; zero executed spec cases) — explicitly not counted as coverage.
- **No coverage was invented for this document.** Every ✅/🟡 cell above cites the specific spec file(s) and, where applicable, the specific manual case ID range it overlaps; every 🔴/⚪ cell states plainly why it is not counted as covered.
